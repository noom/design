App Interface Guidelines

We want to create harmonious experiences for our users at Noom, and we do this by applying solutions consistently across our product suite, and promoting learnability. This section of the guidelines is intended to help standardize mockups and designs for Noom's core application: Noom Coach. This does not include each individual element or interaction in the app, but rather aims to provide the design patterns which make up the core Noom Coach Features. This guideline is be an on-going project, and will be revised and expanded continuously to suit changing product needs.

Introduction

The Noom design system is comprised of a series of page design elements and modules which work together to shape the Noom experience. Use the rules outlined in this design system to create new features or functionality to ensure a consistent user experience. The design team uses specific language when referring to specific locations/modules/screens in the app. The figure below describes these nicknames.

Buttons

Buttons are tactile elements in UI that users can interact with to perform a specific function. Well designed buttons give users foresight into what the intended purpose of the button is prior to the interaction, and provide feedback after the button has been pressed.

Types of Buttons

Text buttons

Text buttons are text only buttons. They lay flat on the same plane as the main interface, and are commonly used in dialogs, nav bars, etc.

Floating buttons

Floating buttons represents the primary action in an application. They are on the highest layer in the hierarchy, and cast a drop shadow on the interface below.

Full buttons

Full buttons are rectangular buttons with rounded corners. They consist of text and a solid colored rectangular background. They are not raised and lay on the same plane as the main interface. Due to mobile behaviour, there are no post-interaction animations associated with these buttons (darkening).

Hierarchies

In situations where decision making is involved on the user's part, with one being the prominent option, button hierarchy is required. The primary action is the button that is considered the preferred action. For example, when a user is trying to delete an exercise entry, "Cancel" would be treated as the primary action, and would be given more visual prominence. In the case of dialog text buttons, the primary action should be on the right hand side of the dialog. In the case of full buttons, the primary action should be described in the blue rectangular button, with the secondary action appearing below as a text button.

Cards

Cards are a convenient means of displaying and organizing content composed of a number of different elements. They serve as an entry point for users to either receive more information or gain access to more interactions with a more detailed panel. Cards may contain an image, text, and actions about the subject.

Usages

Content

Hierarchy should be used within the card to promote legibility. By placing the most important information at the top of the card would draw the user’s attention. Typography should be appropriately used to emphasize the level of informational importance. Background images should not be used in the cards where texts occupy the same space.

Actions

The primary action in a card is typically content exploration or interaction. Cards serve as a “summary window” upon which the executive information is displayed. By tapping on the card the main UI should transform into one in which more details and interactions will be available to the user.

Usages

  • Tapping on the task card allows the user to log their meals
  • Tapping on a recipe card opens up a full activity that provides the user with more actions and details surrounding the recipe

Supplemental actions

Supplemental actions are interactions within the cards that have explicit visual indicators which gives users an overview of their functionality. These actions include text links, progress sliders, selectors, icon buttons, etc. The supplementary actions occupy a separate row of action area from the main interactions. Universally shared actions such as close and back are not considered supplemental actions and could coexist in the same real estate as the main card content.

Dialogs

Dialogs inform users about critical information, and sometimes require decision input, or involve multiple tasks. Dialogs usually contain text, occasionally images, and often buttons or navigation elements. They should occupy the center of an user’s attention until it has been dismissed or a required action has been taken.

Fullscreen modal

Fullscreen modals is a dialog type that acts as an overlay over the entire screen. Commonly used to accommodate for Call-to-Action related screens.

Actions

Fullscreen modals typically contain no internal navigation. Their main usage is to present the users with a large amount of information. A single or at most two call-to-action buttons should appear on the bottom of the fullscreen modal to lead the user away from the modal panel. Links are often included on fullscreen modals that leads the user to external workflows (calorie breakdown analysis).

Usages

  • Coach weight customization quiz flow
  • Program selection quiz flow
  • Meal feedback “achievement” screen
  • Meal reminder screen
  • Recipe cards

Gopher modal

Gopher modals are windows that give users the perception of a drag-down. These modals can be overlapped on top of one-another depending on the workflow requirements. Gopher modals are “anchored” to the bottom of the page. The content of these gopher modals span a very broad range, but the one thing they all have in common is the drag-down button on the top of the modal. This button allows users to dismiss the gopher modals whenever they choose.

Actions

The main purpose of a gopher modal is to present users with high-level information about a particular subject, and ask the user to quickly input decisions based on the prompt given. Due to this behaviour, the content of gopher modals are typically not of high priority, and the input could be delayed based on the user’s discretion.

Usages

  • Exercise tracking
  • Steps counter
  • Meal logging
  • Blood pressure tracking
  • Weight tracking
  • Food logging custom units
  • Group user info

Full activity

Full activities are dialogs that replace the entire screen. These dialogs overrides the current active window.

Usages

  • Noom articles
  • Blood glucose tracking on Android

Toaster notifications

Alert dialog

System texts that provide simple information to the user. Some of these dialog prompts require no interactivity or decisions to be made from the user’s part. Generally try to avoid using these in designs as they are usually reserved for lower priority decisions.

Icons

Illustrations

Infographics

Infographics are visual graphics that digests data and communicates high-level information about the data to the users. They appear in the application in a variety of forms, and are not directly interactable. The appearance of infographics can be altered by changing the parameters related to the graphics, but must be done from outside of the image itself. For design and development purposes, infographics are not directly interactable. The main forms in which the infographics take on are bar graphs, data sliders, and scatter plots.

Bar graph

These infographics appear as stacked bar graphs. Most notable use for this application is in the calories breakdown graphics under the analysis section in food logging. The segments within each bar is represented by the relative amount of calories each food contains compared to the total amount calorie intake.

Data slider

Data sliders are used in a number of locations, each with an altered appearance.

Inputs

The input controls allow users to make a decision or indicate preferences when presented with different options.

Checkboxes

Checkboxes allow users to select multiple options from a list. These are best used for a list of options that could be categorized under a single series. For single options, the on/off switches should be used instead.

Forms

Radio buttons

Radio buttons are used when the users are required to select one option from a set of options. These are best reserved for selecting one option out of a series of mutually exclusive options. Radio buttons are best used for situations where it is important for users to see and compare all the options simultaneously. Segmented controls have similar functionalities as radio buttons as they allow users to select one option out of a series. Radio buttons should be used for lengthier option descriptions, and options with three or more options. For simpler (often one-word) and fewer (less than three) options, segmented controls should be used.

Segmented control

A segmented control is a set of two or more options, each of which functions as a mutually exclusive option. The buttons can contain texts or icons. The selection option should be highlighted in a way that makes intuitive sense to the user. Segmented controls are most appropriately used for simpler options that have short descriptions. For a series of options that contain longer names, radio button options should be opted instead.

Switches

Switches toggle the state of a single settings option. These switches allow users to make binary selections between the On and Off states. The On state for the switches appear illuminated. In the Off state, the switches fade out to show that it is inactive.

Tape

There are two main components to a “tape”. The unit selector and the number scroller. The increments on the number scroller changes dynamically depending on the unit that’s chosen. In a couple of cases however, the unit selector is not required when there is only one recognized unit (year in age selection, and mmHg in blood pressure logging). In most cases the increments for the different units are chosen intuitively. The segments should mimic what one would expect if weights could be measured using a ruler.

Lists

These contain an ordered list of options, items, or tasks. In task lists, the items are ordered in chronological order. In food logging, the items are ordered in alphabetical order. User are organized by alphabetical order in Coach groups. Note that list items can contain texts, icons, interactable components such as checkboxes, or combinations of the three.

Usages

Menus

Menus display a list of choices on a dropdown. They typically appear after interaction with a button. They display a one choice per line on a sheet of card-like material.

Usages

There should at least be three or more items for a menu to be considered appropriately used. If there are two or fewer items in a menu dropdown, consider using icons or links instead to represent the options. Menus should contain links to pages that are secondary to the screen’s function, and should not be the primary way for users to navigate the app. Navigation menus or tabs should be used instead for navigation to important pages.

Navigation

Navigation behavior should be consistent and easily accessible and visible to the user. These elements should be placed in intuitive and predictable locations so new and returning users could move through the app with ease.

Nav bar

The nav bar serves as a hub to house navigation elements such as tabs, menus, and back buttons to allow users to move through the app.

Tabs

Tabs allows users to make horizontal navigations. Tabs offer menu options that are parallel in the hierarchy. They are often used to display different views. For example tabs allows the user to switch between Breakfast, Lunch, and Dinner for recipes selection.

Usages

  • Food logging method selection
  • Recipes menu filter

Back

The back button returns the user vertically up by one level. It navigates upwards in the app’s hierarchy until the home screen is reached.

Close

The close button returns the user to the highest level of vertical hierarchy that is present. Typically it will close the entire work flow and return the user to the home screen.

Usages

Progress indicator

Progress indicators are the UI elements that provides users with an visual indication of the progress in an operation.

Linear

Linear progress are horizontal bars that increase in meter as more values are entered. These bars should never decrease in value.

Usages

  • Calorie tracking
  • Pedometer exercise tracking
